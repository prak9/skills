# Project Control Docs

## 目标

建立一套最小但闭环的 AI 协作文档，使项目同时具备：

- **速度**：AI 在明确边界内自主设计、实现、修复和验证。
- **方向**：人的注意力集中在端到端业务结果，而不是逐行指挥。
- **安全**：通过硬约束、风险通道、Checkpoint、证据和回滚限制损失。

默认只维护三类文档：

```text
program.md                 # 项目级 Source of Truth：意图、边界、决策权
codemap.md                 # 实现世界的当前地图：代码、数据、依赖、Harness
tasks/TASK-NNN-<slug>.md   # 单一任务包：推进、Checkpoint、证据、回滚
```

不要默认创建独立的 `spec.md`、`decisions.md`、`runbook.md` 或 Checkpoint 文档。先将它们合并到上述三类文件；只有达到“何时拆分”条件时才拆。

## 核心模型

### 水流模型

把 AI 的推进视作水流：

- **边界**决定水不能流向哪里。
- **Checkpoint**把推进拆成可观测、可干预的节点。
- **风险通道**决定何时自主推进、何时展示、何时停止。

### 最小混沌单元

每个任务包必须同时满足：

1. 小到目标清楚。
2. 小到边界可见。
3. 小到结果能验。
4. 小到失败能局部回炉或回滚。
5. 大到 AI 仍有自主设计、实现、修复和验证的空间。

原则：**小到可检查，大到可自治。**

### 人的把控价值

人主要判断：

- 端到端效果是否成立。
- 结果是否有足够证据。
- 剩余风险是否被消除、控制或明确接受。

不要把人的主要工作退化为逐步批准普通实现细节。

## 权威与事实

分别处理“意图”和“现实”：

- 目标、范围、约束和授权冲突时：`program.md > 已批准的当前任务包 > 聊天记录`。
- 当前实现行为冲突时：代码、测试和运行证据是事实；`codemap.md` 只是摘要。
- 文档与实现不一致时，标记为漂移并展示，不得默默把一方改写成另一方。
- 聊天中的重要结论，未写入 `program.md` 或任务包前，不视为持久决策。

## 先判断工作模式

### 模式 A：0→1 初始化

适用于想法模糊、没有 `program.md`，或目标尚未形成可执行规格。

### 模式 B：1→N 续接

适用于已有 `program.md`、`codemap.md` 或任务包，需要跨会话继续推进。

### 模式 C：切任务包

适用于目标已明确，需要把一项工作切成 AI 可自治、人可验收的完整单元。

### 模式 D：审计与修复

适用于文档膨胀、互相冲突、已过期、缺少证据，或需要合并现有 Markdown。

## 通用工作流

1. 读取用户提供的信息和仓库中已有的相关文档。
2. 不重复询问已经有答案的问题。
3. 检查代码、测试、配置、运行入口和最近变更；不要仅凭文件名猜测。
4. 判断工作模式。
5. 只加载当前需要的模板：
   - 创建或重构项目控制面时，读取 `assets/program.template.md`。
   - 创建或刷新实现地图时，读取 `assets/codemap.template.md`。
   - 创建任务包时，读取 `assets/task.template.md`。
6. 写入或更新文件。
7. 执行“质量检查”；环境允许时运行 `scripts/validate_docs.py <项目根目录>`。
8. 汇报改动、证据、未决事项和下一 Checkpoint。

## 模式 A：0→1 初始化流程

### 1. 先做理解回显

在高成本实施前，先用一屏以内复述：

- 业务结果：最终要改变什么。
- 成功指标：怎样算成功，怎样算完成。
- 硬约束与不可接受的失败。
- 非目标与关键假设。
- 软偏好的优先级。
- AI 可自主决定与必须先征询的边界。
- 所需验收证据。
- 迁移、回滚和未来替换方式。

发现问题时按以下方式处理：

- 偏了：纠正目标。
- 空了：补证据和度量。
- 宽了：收边界和非目标。
- 未验证：标记 `[待验证]` 并给出验证方法。
- 必须由人取舍：标记 `[待决策]`，集中展示少量实质选项。

不要用连续追问阻塞可逆工作。用户要求直接生成时，基于现有信息创建明确标注的草案。

### 2. 把搜索空间压进 program.md

使用 `assets/program.template.md`，只写项目级且跨任务仍成立的内容：

- 业务结果和可度量成功标准。
- 范围、关键场景、非目标和假设。
- 硬约束、不可接受失败和软偏好排序。
- 决策权、风险通道和项目级验收证据。
- 迁移、回滚、关键决策和会话续接协议。

### 3. 建立 codemap.md

- 对绿地项目，明确标记“尚无实现”，仅记录预定入口和待验证结构。
- 对已有仓库，只记录已经观察到的实现事实。
- 记录版本或 commit，防止把旧地图误当当前现实。

### 4. 切第一个任务包

第一个任务包必须只解决一个问题，并优先消除最高风险假设或建立最短端到端证据链。

## 模式 B：1→N 续接流程

开始前依次读取：

1. `program.md`
2. `codemap.md`
3. `program.md` 指向的当前任务包
4. 最近相关提交、测试结果和证据

随后给出简短续接摘要：

- 当前业务目标和硬边界。
- 当前任务状态与下一 Checkpoint。
- 文档与代码是否漂移。
- 尚未验证的假设和剩余风险。
- 本轮将在授权范围内自主推进的事项。

若 `codemap.md` 与当前代码明显不符，先修正地图或标记漂移，再依赖它规划工作。

## 模式 C：任务包规则

任务包不是待办列表，而是一个完整单一问题的控制单元。

必须包含：

- 一个可观察的结果，而不只是“完成若干代码改动”。
- 明确的要做、不做、允许修改和禁止修改范围。
- 可验证的验收标准及证据来源。
- AI 自主空间和本任务特有的升级条件。
- 在仍可干预的位置设置 Checkpoint。
- 风险、影响半径、回滚触发和恢复方法。
- 完成时回写 `program.md` 与 `codemap.md` 的要求。

不要把互不依赖的多个结果塞进一个任务包。若一个部分失败后无法局部回炉，应继续拆分。

任务文件命名：

```text
tasks/TASK-001-short-slug.md
```

编号递增；已关闭任务不复用编号。

## 模式 D：审计与修复规则

审计时检查：

- 是否存在多个文件争夺同一事实的权威。
- `program.md` 是否混入大量易变实现细节。
- `codemap.md` 是否写了计划而不是当前事实。
- 任务包是否变成长期 backlog 或多目标集合。
- 成功指标是否只有形容词，没有目标值和观察方式。
- 每条硬约束是否有对应验收证据。
- 每个不可接受失败是否有预防、检测和限制影响的方法。
- 是否把“代码完成”误当成“业务结果成立”。
- 当前任务、下一 Checkpoint、证据链接是否失效。

修复时优先删除重复、建立引用和恢复唯一权威，不要机械保留所有历史文字。

## 风险通道

### 绿灯：自主推进

同时满足以下条件时，AI 可自行决定并继续：

- 局部且可逆。
- 不改变外部契约。
- 不触碰硬约束。
- 失败可被测试发现并局部回炉。
- 任务包已授权该类实现选择。

### 黄灯：到 Checkpoint 展示

出现以下情况时，在继续扩大影响前展示：

- 跨模块或跨任务包影响。
- 关键假设仍未验证。
- 引入长期技术承诺、显著成本或迁移负担。
- 多个合理方案在重要偏好上存在明显取舍。
- 验收证据不足，但仍可安全进行有限探索。

### 红灯：停止并先征询

出现以下情况时，不得继续实施：

- 可能违反安全、数据、隐私、合规或兼容性硬约束。
- 可能造成不可逆数据损失、越权或大范围中断。
- 需要改变业务目标、非目标或项目级验收标准。
- 需要改变外部 API、持久化格式、权限或用户可见行为，且未获授权。
- 回滚路径不存在或未经验证，而操作具有较大影响半径。

## Checkpoint 契约

每个 Checkpoint 必须回答：

1. **结果**：当前已经得到什么可观察结果。
2. **变化**：相比上个 Checkpoint 改变了什么。
3. **证据**：测试、基准、监控、演练或人工验证在哪里。
4. **风险**：剩余风险、未知项和影响范围。
5. **决策**：是否需要人选择；若需要，给出少量选项、取舍和建议。
6. **下一步**：下一段自主推进内容和下一 Checkpoint。

不要用进度百分比替代证据。

## 编写规则

- 使用短句、明确动词和可测试条件。
- “提升、稳定、快速、合理、兼容”等词必须配度量或判定方法。
- 项目级稳定事实写入 `program.md`；实现事实写入 `codemap.md`；单次推进写入任务包。
- 同一事实只保留一个权威位置，其他文件用相对链接引用。
- 证据正文可以放在测试报告、日志或产物中；Markdown 只保留摘要和定位信息。
- 使用 ISO 日期 `YYYY-MM-DD`，并在 `codemap.md` 记录 commit/tag。
- 未确认内容统一标记：`[待确认]`、`[待验证]`、`[待决策]`。
- 不要静默改写已批准决策。新增决策条目，明确替代或废弃旧决策。
- 不以模板完整为目标；删除不适用章节或明确写 `不适用：<原因>`。
- 不把计划写成已经发生的事实。

## 角色映射

存在对应角色时：

- **TL**：维护业务结果、范围、优先级和验收契约。
- **架构师**：设计协作姿态、边界、控制面和 Harness 工具集。
- **一线执行者**：切任务包、设置 Checkpoint、审查证据和处理升级。
- **QA**：设计测试策略、失败场景、证据标准和回归范围。
- **AI**：在授权边界内自主设计、实现、测试、修复和报告。

角色不存在时不要虚构姓名；在文档中保留职责并标记 Owner。

## 关闭任务包

任务完成前必须：

1. 运行任务级验收测试。
2. 给出端到端结果证据，而不只给出代码 diff。
3. 验证回滚或说明为何不适用。
4. 将跨任务仍成立的决定写回 `program.md`。
5. 按实际代码更新 `codemap.md`。
6. 更新 `program.md` 的当前任务和下一 Checkpoint。
7. 在任务包中记录剩余问题，不把它们伪装成已完成。

## 何时拆分更多 Markdown

默认不拆。只有出现以下情况时才建议拆分：

- `program.md` 已明显妨碍检索和维护，例如超过约 500 行。
- 关键决策数量较多且需要独立审计，拆为 `decisions/ADR-NNN.md`。
- 运维流程需要脱离开发上下文被独立执行，拆为 `runbook.md`。
- 项目包含多个相对独立产品域，各自有不同目标和验收契约。
- 证据体积大，拆到 `evidence/`，原文档只保留摘要和链接。

拆分后仍须在 `program.md` 保留索引和权威关系。

## 质量检查

完成写作后确认：

- [ ] 一句话能说明业务结果。
- [ ] 成功指标可测，完成定义可判定。
- [ ] 硬约束和不可接受失败没有被弱化。
- [ ] 非目标能阻止范围自然膨胀。
- [ ] 假设有状态和验证方法。
- [ ] 软偏好有明确优先级，可用于取舍。
- [ ] AI 自主权与必须先征询事项不重叠。
- [ ] 每条验收标准都有证据来源。
- [ ] 当前任务是单一结果，失败可局部回炉。
- [ ] Checkpoint 位于仍可干预的位置。
- [ ] 回滚有触发条件，而不只是“可以回滚”。
- [ ] `codemap.md` 基于当前实现并注明版本。
- [ ] 文件之间没有复制粘贴式重复权威内容。
- [ ] 未决项被显式标记，没有被猜测填充。

## 最终汇报格式

完成后只需报告：

```md
## 文档变更
- 新建/更新：<文件>

## 已固定的控制面
- <最重要的目标、边界或授权变化>

## 证据与检查
- <验证命令或检查结果>

## 未决事项
- <只有真正需要人的决定才列出；没有则写“无”>

## 下一 Checkpoint
- <触发条件或位置>
```
